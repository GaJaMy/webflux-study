# Repository Guidelines

## Project Structure & Module Organization

This repository is a Gradle multi-module Spring Boot study project comparing MVC and WebFlux.

- `mvc-api/src/main/java/com/gajamy/webflux/`: Spring MVC baseline source code.
- `mvc-api/src/test/java/com/gajamy/webflux/`: MVC tests mirroring the main package structure.
- `webflux-api/src/main/java/com/gajamy/webflux/webfluxapi/`: minimal WebFlux application skeleton. Add WebFlux features here step by step.
- `webflux-api/src/test/java/com/gajamy/webflux/webfluxapi/`: WebFlux tests.
- `load-test/`: k6 scripts used to compare API behavior under load.
- `docs/`: study notes and tool usage guides.
- `build.gradle` and `settings.gradle`: Gradle build and project metadata.
- `gradle/`, `gradlew`, `gradlew.bat`: Gradle wrapper files. Use the wrapper instead of a system Gradle install.

## Build, Test, and Development Commands

Run commands from the repository root:

```bash
./gradlew :mvc-api:bootRun
```

Starts the MVC application locally.

```bash
./gradlew :webflux-api:bootRun
```

Starts the WebFlux application locally on port `8081`.

```bash
./gradlew test
```

Runs the JUnit Platform test suite.

```bash
./gradlew build
```

Compiles the project, runs tests, and creates build artifacts.

```bash
./gradlew clean
```

Removes generated build output.

## Coding Style & Naming Conventions

Use Java 17 and the conventions already present in the project. Java source uses tabs for indentation as generated by the current Gradle/Spring initializer style. Keep package names lowercase, classes in `PascalCase`, methods and fields in `camelCase`, and constants in `UPPER_SNAKE_CASE`.

Prefer constructor injection for Spring components. Use Lombok only where it reduces boilerplate without hiding important behavior. Keep configuration in `application.yaml`; do not hard-code environment-specific values.

## Testing Guidelines

Tests use Spring Boot Test and JUnit 5 via `useJUnitPlatform()`. Name test classes after the unit or feature under test, for example `OrderServiceTest` or `ProductControllerTest`.

Place tests under the matching module's `src/test/java`. For narrow logic, prefer plain unit tests. Use `@SpringBootTest` only when the Spring context is needed. Run `./gradlew test` before submitting changes.

## Commit & Pull Request Guidelines

No Git history is available in this working tree, so use a simple, consistent commit style:

```text
feat: add product lookup endpoint
fix: handle missing order id
test: add controller validation tests
```

Pull requests should include a short description, the reason for the change, test results such as `./gradlew test`, and any relevant API examples. Link issues when applicable. For behavior changes, include request/response examples or screenshots from API tools if helpful.

## Security & Configuration Tips

Do not commit secrets, credentials, or local machine paths. Put local overrides in ignored files or environment variables. Keep `application.yaml` safe for shared development defaults.

## Study Continuation Notes

Before continuing the MVC/WebFlux study flow, read `docs/study-checkpoint.md`. It records the implemented endpoints, k6 results, current conclusions, and the next planned experiment.
