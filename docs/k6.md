# k6 Usage Guide

이 문서는 Spring MVC/WebFlux 성능 비교 학습을 위한 k6 설치 및 실행 방법을 정리합니다.

## k6란?

k6는 JavaScript로 부하 테스트 시나리오를 작성하고 CLI에서 실행하는 성능 테스트 도구입니다. 이 프로젝트에서는 `/delay` API에 동시 요청을 보내고 응답 시간, 처리량, 실패율을 비교하는 데 사용합니다.

## macOS 설치

Homebrew를 사용하는 경우:

```bash
brew install k6
```

설치 확인:

```bash
k6 version
```

Spring Boot 서버 실행:

```bash
./gradlew :mvc-api:bootRun
```

다른 터미널에서 k6 실행:

```bash
k6 run load-test/delay-smoke.js
```

## Windows 설치

winget을 사용하는 경우:

```powershell
winget install k6 --source winget
```

Chocolatey를 사용하는 경우:

```powershell
choco install k6
```

설치 확인:

```powershell
k6 version
```

Spring Boot 서버 실행:

```powershell
.\gradlew.bat :mvc-api:bootRun
```

다른 터미널에서 k6 실행:

```powershell
k6 run load-test/delay-smoke.js
```

## Docker로 실행

로컬에 k6를 설치하지 않고 Docker로 실행할 수도 있습니다.

macOS/Linux:

```bash
docker run --rm -i grafana/k6 run - < load-test/delay-smoke.js
```

Windows PowerShell:

```powershell
Get-Content load-test/delay-smoke.js | docker run --rm -i grafana/k6 run -
```

## 테스트 스크립트 위치

k6 스크립트는 다음 위치에 둡니다.

```text
load-test/
├── delay-smoke.js
└── delay-load.js
```

권장 순서:

1. `delay-smoke.js`: 낮은 부하로 API가 정상 동작하는지 확인
2. `delay-load.js`: 동시 사용자 수를 늘려 MVC/WebFlux 성능 비교

## k6 기본 사용법

k6 스크립트는 JavaScript 파일입니다. 일반적인 구조는 다음과 같습니다.

```js
// 1. 필요한 k6 모듈 import
// 2. options로 부하 조건 설정
// 3. default function에 반복 실행할 요청 작성
// 4. check 또는 thresholds로 성공 조건 정의
```

가장 중요한 개념은 다음과 같습니다.

- `VU`: Virtual User. 동시에 요청을 보내는 가상 사용자입니다.
- `duration`: 테스트를 얼마나 오래 실행할지 정합니다.
- `stages`: VU 수를 시간에 따라 늘리거나 줄이는 설정입니다.
- `default function`: 각 VU가 반복해서 실행하는 함수입니다.
- `check`: 응답 상태나 값이 기대와 맞는지 확인합니다.
- `thresholds`: 테스트 성공/실패 기준입니다. 예를 들어 p95 응답 시간이 1500ms 미만이어야 한다는 조건을 둘 수 있습니다.

간단한 실행 명령은 다음과 같습니다.

```bash
k6 run load-test/delay-smoke.js
```

스크립트 파일을 고치지 않고 CLI에서 VU와 시간을 지정할 수도 있습니다.

```bash
k6 run --vus 10 --duration 30s load-test/delay-smoke.js
```

이 명령은 10명의 가상 사용자가 30초 동안 `default function`을 반복 실행한다는 의미입니다.

## k6 스크립트 구성 요소

### http 요청

HTTP API를 호출할 때는 `k6/http` 모듈을 사용합니다.

```js
// TODO: http.get으로 GET 요청을 보낸다.
// TODO: http.post로 POST 요청을 보낸다.
// TODO: 응답 객체에서 status, body, timings 등을 확인한다.
```

### sleep

`sleep`은 각 반복 사이에 쉬는 시간을 둡니다.

```js
// TODO: sleep(1)을 사용하면 각 VU가 요청 후 1초 대기한다.
// TODO: sleep이 없으면 가능한 한 빠르게 요청을 반복한다.
```

성능 한계를 보고 싶을 때는 sleep을 줄이거나 제거할 수 있습니다. 실제 사용자 행동을 흉내 내고 싶을 때는 sleep을 둡니다.

### checks

`check`는 개별 응답이 기대 조건을 만족하는지 확인합니다.

```js
// TODO: 응답 status가 200인지 확인한다.
// TODO: 응답 body에 특정 값이 있는지 확인한다.
```

check가 실패해도 기본적으로 테스트 실행은 계속됩니다. 실패율은 결과에 표시됩니다.

### thresholds

`thresholds`는 전체 테스트의 성공 기준입니다.

```js
// TODO: http_req_failed 실패율을 1% 미만으로 제한한다.
// TODO: http_req_duration p95를 1500ms 미만으로 제한한다.
```

threshold가 실패하면 k6 명령이 실패 상태로 종료됩니다. 성능 기준을 자동으로 검증할 때 유용합니다.

## 부하 테스트 종류

처음부터 강한 부하를 주지 말고 아래 순서로 진행합니다.

| 종류 | 목적 | 예시 |
|---|---|---|
| Smoke test | API가 부하 테스트 환경에서 정상 동작하는지 확인 | 1 VU, 10초 |
| Load test | 예상 사용량 수준에서 성능 확인 | 10~100 VU, 30초~5분 |
| Stress test | 한계를 찾기 위해 점진적으로 부하 증가 | 100 VU 이상 |
| Spike test | 갑작스러운 트래픽 증가 확인 | 짧은 시간에 VU 급증 |

이 프로젝트에서는 먼저 Smoke test와 Load test만 사용합니다.

## 첫 Smoke Test 시나리오

`load-test/delay-smoke.js`에는 다음 내용을 구현합니다.

```js
// TODO: k6/http에서 http import
// TODO: k6에서 sleep import
// TODO: options 정의
//   - vus: 1
//   - duration: '10s'
// TODO: default function 작성
//   - GET http://localhost:8080/delay?ms=100
//   - 응답 상태가 200인지 check
//   - sleep(1)
```

실행:

```bash
k6 run load-test/delay-smoke.js
```

## 결과에서 볼 지표

k6 실행 결과에서 우선 다음 지표를 확인합니다.

- `http_req_duration`: 요청 응답 시간
- `http_req_failed`: 실패율
- `http_reqs`: 전체 요청 수와 초당 요청 수
- `iterations`: 시나리오 반복 횟수
- `vus`: 사용된 가상 사용자 수

특히 비교 기록에는 다음 값을 남깁니다.

- `avg`
- `p(95)`
- `p(99)`
- `http_reqs` rate
- `http_req_failed`

예를 들어 `/delay?ms=1000`을 테스트하면 정상적인 응답 시간은 최소 1000ms 근처입니다. MVC에서 `Thread.sleep(1000)`을 사용하기 때문에 요청 스레드는 그 시간 동안 대기합니다.

## 결과 해석 방법

처음에는 모든 지표를 보려고 하지 말고 아래 순서로 봅니다.

1. `http_req_failed`가 0%에 가까운지 확인합니다.
2. `http_req_duration`의 `avg`, `p(95)`, `p(99)`를 확인합니다.
3. `http_reqs`의 초당 요청 수를 확인합니다.
4. VU를 늘렸을 때 실패율이나 p95가 급격히 증가하는지 확인합니다.

`avg`만 보면 일부 느린 요청을 놓칠 수 있습니다. 그래서 비교할 때는 `p(95)`와 `p(99)`를 함께 봅니다.

## 이 프로젝트의 첫 실행 순서

1. Spring Boot 서버를 실행합니다.

```bash
./gradlew :mvc-api:bootRun
```

Windows:

```powershell
.\gradlew.bat :mvc-api:bootRun
```

2. 다른 터미널에서 k6 smoke test를 실행합니다.

```bash
k6 run load-test/delay-smoke.js
```

3. 문제가 없으면 CLI 옵션으로 VU를 늘려 봅니다.

```bash
k6 run --vus 10 --duration 30s load-test/delay-smoke.js
```

4. `/delay?ms=1000` 기준의 정식 load test는 `delay-load.js`로 분리합니다.
